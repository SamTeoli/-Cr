# -Cr

카드 수집형 덱빌딩 로그라이크 **Have a Break, and then..**

## 카드 데이터 시작하기

1. Unity에서 프로젝트를 연다.
2. 상단 메뉴에서 `Have a Break > Card Data Tools`를 선택한다.
3. 카드 종류, 수록카드ID, 이름, 등급, 마력 코스트를 입력하고 `Create Card Asset`을 누른다.
4. 생성된 카드의 상세 수치와 효과를 Inspector에서 편집한다.
5. `Validate Cards`로 누락·중복 ID를 검사한다.
6. `Rebuild Card Database`로 `Assets/GameData/CardDatabase.asset`을 갱신한다.

카드 에셋은 기본적으로 `Assets/GameData/Cards`에 생성된다. 카드 종류는 몬스터, 스킬,
트랩, 결계이며 등급은 일반, 희귀, 전설이다. 코드의 영문 enum 이름 `Barrier`는 게임 내
표시 용어인 **결계**에 대응한다.

ID 계층은 수록카드ID(`CatalogCardId`), 보유카드ID(`OwnedCardId`),
전투카드ID(`BattleCardId`), 전투 중 임시 생성·복제용 `InstantId` 순으로 분리한다.

## 시험 카드 C01~C12

`Assets/GameData/Cards`에 Google Docs `카드 모음집 · C01~C12` v0.4를 기준으로 한
시험 카드 12종이 등록되어 있다. 각 에셋은 기본 카드 정보와 레벨 1~5의 비용·공격력·
생명력·효과 문구를 보유한다. `CardDatabase.asset`에도 C01부터 C12까지 ID 순으로 연결되어
있다. 효과 문구는 기획 원문을 보존한 데이터이며 실제 전투 효과 실행기는 후속 단계에서
연결한다.

`BattlePlayerState`는 기본 최대 체력 30과 전투 중 현재 체력을 관리한다. 최대 체력이
증가하면 현재 체력도 같은 양만큼 증가하고, 감소할 때는 현재 체력이 새 최대치를 넘는
경우에만 맞춘다. 피해·회복 실행기는 `PLAYER` 대상을 몬스터와 구분해 같은 사건 구조로
기록한다. `BattleEnemyTracker`는 살아 있는 적 ID를 관리하며 `BattleOutcomeEvaluator`는
플레이어 체력 0을 먼저 검사한 뒤 남은 적이 없으면 승리로 판정하므로 동시 승패에서는
패배가 우선한다.

`BattleSettlementService`는 승패가 확정되고 효과 대기열이 비었을 때만 전투 종료를
반영한다. 배틀의 최종 플레이어 체력, 실제 소비한 소모아이템과 전투 중 골드 증감을
`RunBattleState`에 원자적으로 한 번만 적용한다. 패배하면 런 종료 상태가 되며 보상 대상이
아니다. 승리는 보상 가능 상태만 표시하고 실제 승리 보상 지급은 이후 보상 단계에 맡긴다.
반영 완료 뒤 배틀 상태는 별도 사건을 만들지 않고 폐기된 것으로 표시한다.

`BattleVictoryRewardService`는 전투 종료 반영이 끝난 승리만 보상 대상으로 인정한다.
일반 전투 골드는 20·25·30, 엘리트는 40·45·50·55 중 시드로 동일 확률 선택하며
중간보스는 60, 최종 보스는 런 골드를 지급하지 않는다. 보상 골드는 런 상태에 한 번만
지급한다. 인첸트 선택지 수, 희귀 이상 보장과 엘리트 소모아이템 수는 보상 구성 정보로
제공하며 실제 인첸트·아이템 후보 생성은 해당 데이터 구조가 연결된 뒤 처리한다.

`BattleVictoryEnchantRewardService`는 승리 반영 뒤 외부에서 전달한 인첸트 선택지를 전투
등급별 개수, 중복 ID, 등급 보장과 현재 런 덱의 즉시 장착 가능 여부로 검증한다. 후보
확률이나 생성 규칙은 임의로 만들지 않으며, 검증된 선택지 중 하나만 보유카드ID와 슬롯
번호를 지정해 장착할 수 있다. 장착 직전에도 카드 호환성, 빈 슬롯과 중복 제한을 다시
확인한다.

`RunEncounterProgressState`와 `RunEncounterProgressService`는 런 상태·런 덱과 현재 전투를
함께 관리한다. 활성 전투가 있거나 이미 사용한 전투 인스턴스ID이면 새 전투를 시작하지
않으며, 승리 뒤 골드·인첸트·소모아이템 보상이 남아 있으면 다음 전투로 진행하지 않는다.
최종 보스 영구 보상이 남아 있을 때도 완료 처리하지 않고 별도 대기 사유를 반환한다.
패배 정산을 완료하면 활성 전투를 닫고 런 종료 상태를 유지한다.

`BattleVictoryConsumableRewardService`는 엘리트 승리 정산 뒤 외부에서 확정해 전달한
소모아이템ID를 보상 수량만큼 런 상태에 추가한다. 아이템 종류, 후보와 확률은 임의로
생성하지 않으며 빈 ID, 전투당 중복 서비스 생성과 요구 수량을 넘는 중복 수령을 차단한다.
소모아이템 보상 수령을 마치면 런 진행 서비스의 해당 대기 상태가 해제된다.

`PlayerPermanentRewardState`는 런 전투 상태와 분리된 영구 보상ID 목록을 관리하며 새 런을
만들 때 같은 상태를 다시 주입할 수 있다. `BattleVictoryPermanentRewardService`는 최종
보스 승리 정산 뒤 외부에서 확정해 전달한 영구 보상ID를 한 번만 등록한다. 보상 종류,
효과, 후보와 확률은 임의로 만들지 않으며 빈 ID, 중복 서비스 생성과 이미 보유한 ID의
중복 등록을 차단한다. 영구 보상 수령을 마치면 최종 보스 전투의 진행 잠금이 해제된다.

`EnchantData`는 인첸트 정의 ID, 이름, 등급, 호환 카드 종류와 같은 카드 중복 허용 여부를
가진다. `RunCardEnchantState`는 카드별 기본 슬롯 1칸과 최대 3칸, 실제 장착 순서와 활성
상태를 런 동안 관리한다. 보상·상점 인첸트는 빈 슬롯과 호환성을 최종 재검사한 뒤 즉시
장착하며 전투 중 장착·제거는 차단한다. 제거해도 다른 인첸트는 당겨지지 않으며, 카드
종류가 일시적으로 비호환이 되면 슬롯을 유지한 채 비활성화하고 다시 호환되면 활성화한다.

`Assets/GameData/Enchants`에는 Google Docs 시험 인첸트 제작서 v0.4를 기준으로 한
E01~E08 정의가 저장된다. 각 정의는 적용 유형, 역할, 추가 호환 조건, 화면 문구, 원본
문서와 효과 판정 목록을 보존하며 `EnchantDatabase.asset`에서 ID 순으로 조회한다.
Editor 도구의 `Create/Rebuild Test Enchants E01-E08`로 에셋을 생성·갱신하고
`Validate Test Enchants E01-E08`로 원문 이전 결과와 데이터베이스 연결을 검사한다.

C01~C12는 카드 원문을 변경하지 않는 구조화된 인첸트 호환 태그를 가진다.
`EnchantCompatibilityEvaluator`는 카드 종류뿐 아니라 E01의 기본 생명력, E02의 기본
공격, E03의 본 효과 완료, E04의 수록 기본 비용, E05의 적 영향, E06의 수치형 반복,
E07의 정상 묘지 이동과 E08의 적 1개 고정 대상 조건까지 검사한다. Editor 도구에서
태그 적용 후 전체 E01~E08 × C01~C12 호환 행렬을 검증할 수 있다.

`CardData.ResolveLevel(level)`은 요청 레벨을 1~5로 보정하고 해당 레벨의 비용·공격력·
생명력·효과 문구를 `ResolvedCardData`로 반환한다. Editor 도구의
`Validate Level Resolution`으로 대표 카드와 범위 보정을 검사할 수 있다.

`BattleCardInstance`는 전투 중 카드의 원본 수록카드, 보유카드ID, 전투카드ID,
인스턴트ID, 현재 레벨과 현재 영역을 관리한다. 전투 중 생성·복제된 카드는
`InstantId`로 임시 카드임을 구분한다.

`BattleCardZoneState`는 전투 카드 인스턴스를 전투카드ID로 한 번만 등록하고 카드 영역별
조회와 이동을 담당한다. 패는 10장, 몬스터 필드와 스킬 필드는 각각 3장으로 제한하며,
용량 초과와 중복 전투카드ID는 상태를 변경하지 않고 실패 사유를 반환한다.

`BattleDeckState`는 전투 시작 시 시드 기반으로 드로우 더미를 섞고 시작 패 5장을
뽑는다. 플레이어의 첫 턴 드로우는 건너뛰며 이후 턴부터 1장씩 뽑는다. 드로우 더미가
비면 묘지를 같은 난수 상태로 다시 섞어 드로우 더미로 옮긴다. 패가 10장이면 카드를
소모하지 않고 `HandFull`을 반환한다. Editor 도구의 `Validate Battle Deck Draw`로
초기 패, 첫 턴 예외, 패 상한, 묘지 재구성과 동일 시드 재현성을 검사할 수 있다.

`BattleCardPlayState`는 카드 사용의 미리보기와 최종 확정을 분리한다. 확정 직전에 패,
마력, 필드 공간과 같은 이름 결계 중복을 다시 검사하며 실패하면 마력과 카드 위치를
변경하지 않는다. 몬스터는 몬스터존에 소환되고, 스킬은 스킬존을 임시 점유한 뒤 묘지로
이동하며, 트랩과 결계는 스킬존에 유지된다. 임시 스킬이 묘지로 이동하려 하면 소멸
영역으로 대체된다. `BattleManaState`는 기본 최대 마력 5, 시스템 상한 10, 플레이어 턴
시작 충전과 턴 종료 제거를 관리한다.

`BattleDeckState.TryRedrawStartingHand`는 시작 패에서 선택한 0~5장을 전투당 한 번
교체한다. 선택 카드는 `RedrawHolding`에 임시 분리하고 남은 드로우 더미에서 같은 수만큼
먼저 받은 뒤, 분리한 카드를 되돌려 다시 섞는다. 교체 카드는 일반 드로우로 처리하지
않으며 교체 완료 뒤 첫 플레이어 턴의 일반 드로우 생략 상태는 그대로 유지된다.

`BattleTurnState`는 전투 구성, 시작 패 교체, 플레이어 행동, 행동 처리 잠금과 적 턴을
순서대로 전환한다. 첫 플레이어 턴은 일반 드로우를 생략하고, 플레이어 턴 종료에는
패를 유지한 채 남은 마력을 0으로 만든다. 적 턴이 끝나면 플레이어 턴 번호를 증가시키고
마력을 최대치로 설정한 뒤 1장 드로우한다. 패가 가득 찼거나 덱과 묘지가 비어 드로우가
실패해도 실패 사유를 기록하고 플레이어 행동 단계에는 정상 진입한다.

`BattleEventLog`는 이미 발생한 사건에 순차 사건ID와 부모 사건, 원인, 출처, 주체,
대상, 영역·수치 변화와 난수 상태를 기록한다. `BattleEffectQueue`는 앞으로 처리할 효과
명령을 사전 수정, 대응, 본 효과, 사후 유발 단계로 분리한다. 대응은 등록 역순으로,
사후 유발 효과는 체계 필수부터 정리 효과까지의 우선순위와 필수·선택·생성 순서로
처리한다. 같은 효과ID는 같은 사건에 기본 한 번만 등록되며 자기 효과가 만든 같은 종류
사건의 재발동은 제한 자료가 없는 경우 차단한다.

`BattleEffectExecutor`는 대기열에서 다음 명령을 꺼내 원본 사건을 확인한 뒤 실행한다.
현재 구현 범위는 패·필드 카드의 영역 이동이며, 성공한 이동만 부모 사건과 전후 영역을
가진 `CardMoved` 사건으로 기록한다. 공간 부족이나 잘못된 대상은 카드를 원래 위치에
남기고 새 사건을 만들지 않는다. 임시 카드의 묘지 이동은 소멸로 대체한다. 드로우 더미
위·아래 위치가 필요한 이동과 드로우·마력·상태 명령은 전용 규칙이 연결될 때까지 명시적인
실패 사유를 반환한다.

`BattleMonsterState`는 원본 카드와 별도로 전투 중 공격력, 최대 체력과 현재 체력을
관리하며 `BattleMonsterRegistry`가 전투카드ID로 상태를 찾는다. 피해와 회복 명령은
현재 체력을 0~최대 체력 범위로 보정하고 적용 전후 수치를 `DamageApplied` 또는
`HealingApplied` 사건에 기록한다. 체력이 0이 된 몬스터는 파괴 후보로 표시하지만 실제
묘지 이동은 후속 상태 기반 검사 단계에서 처리한다. 방어·취약·면역과 초과 피해도 이후
피해 보정 단계에서 연결한다.

`BattleStateBasedChecker`는 원자적 효과가 끝난 뒤 몬스터존을 안정된 필드 목록 순서로
검사한다. 현재 체력이 0인 일반 몬스터는 묘지, 임시 몬스터는 소멸로 이동시키고
`MonsterDestroyed` 사건을 원인이 된 부모 사건에 연결한다. 파괴된 몬스터는 전투 상태
레지스트리에서 제거하며, 이미 처리된 후보를 다시 검사해도 중복 사건을 만들지 않는다.
적 필드와 명시적인 왼쪽·중앙·오른쪽 슬롯 순서는 적 배치 시스템 단계에서 같은 검사기에
연결한다.
